## 1. 概述

两张表都是 CP-SAT 排产求解完成后的**后处理产物**，不参与模型约束，只是对求解出来的 `l[j, t]` (订单 j 在第 t 天占用的产线数) 做"工序级"展开。

| 表 | 视角 | 输出 Sheet 名 |
|---|---|---|
| 表3 | 按订单 × 日期 展开，每行 = 一个 (订单, 日期) 的机台数 | `表3_工序机台数明细` |
| 表4 | 按日期分块，每块包含 当天所有活跃订单 + M汇总 + A理论 + 差异 | `表4_按日期机台数汇总` |

两个 Sheet 名定义在 `config.py:168, 171`。

## 3. 算法核心：`compute_actual_machines(line_count)`

**位置**：`scheduler_results/machine_allocation.py:65-134`

**输入**：单个标量 `line_count`（某订单在某天占用的产线数，0 ~ 18 的整数）

**输出**：长度为 12 的整数列表 `Actual[0..11]`，对应 12 个工序的机台数

**步骤**：

```
1. 工序 12 (丝网) 是产线终点：
   Actual[丝网] = line_count            ← 直接赋值，不查矩阵 A

2. 计算"产线日产能"基准：
   Capacity[final] = line_count × Cj[丝网]   # 丝网 Cj 来自 PROCESS_CAPACITIES[-1]

3. 对工序 1..11，逐个计算：
   raw       = Capacity[final] / Cj[p]
   candidate = ceil(MACHINE_RATIO × raw)     # MACHINE_RATIO = 1.3
   Actual[p] = min(UpperBound[p], candidate) # UpperBound 来自 MATRIX_A

4. 兜底校验（保证产能瓶颈）：
   若 Actual[p] × Cj[p] ≤ Capacity[final]：
       Actual[p] = floor(Capacity[final] / Cj[p]) + 1
       # 这一步可能突破 MATRIX_A 上限
```

**关键设计说明**：

- **为什么丝网是终点不是"成品"**：老师明确反馈"成品"工序在产线上不存在，丝网就是最后一道工序，机台数 = 占用产线数。改动历史见 git log 最近一次提交。
- **为什么需要 `MACHINE_RATIO = 1.3`**：实际生产有换型、停机、损耗，给 30% 缓冲。
- **为什么有兜底**：如果某工序 Cj 太低 + 矩阵 A 上限给得不够，强行卡上限会让丝网产能跟不上。兜底会**突破上限**，由现场处理。

## 4. 配置（要改产线参数就改这几个地方）

所有跟工序机台数相关的常量都集中在 `config.py:106-166`：

| 常量 | 位置 | 当前值 | 含义 |

**输出 DataFrame 结构**（16 列）：

| 列名 | 类型 | 说明 |
|---|---|---|
| 日期 | str | `m/d`，同一日期块内重复 |
| 类型/订单 | str | 订单名 / `M矩阵汇总` / `A矩阵理论(L=X)` / `差异(M-A)` |
| 占用产线 | int | 订单行=该订单产线数；汇总行=当天总产线数；差异行=空 |
| 工序1-制绒 | int | 机台数 |
| ... | ... | ... |
| 工序12-丝网 | int | 机台数 |
| 备注 | str | M汇总行：当天订单数；A理论行：固定提示；差异行：文字总结哪些工序不够/空余 |

**每个日期的块结构**：

```
日期X | 订单A      | 7  | …机台… |
日期X | 订单C      | 7  | …机台… |
日期X | 订单E      | 4  | …机台… |
日期X | M矩阵汇总  | 18 | 上面 3 行求和 | "当天共 3 个订单开线"
日期X | A矩阵理论  | 18 | 查 MATRIX_A[17] | "按当天总产线数从矩阵 A 查表"
日期X | 差异(M-A)  |    | M汇总 - A理论    | "机台不够: 工序X+N…; 空余: 工序Y-M…"
日期X | (空行分隔)
```

**辅助函数**：
- `_build_a_reference_row(total_lines)`：`machine_allocation.py:250-263`，根据当天总产线数查 `MATRIX_A` 得到 A 理论行。
- `_summarize_diff(m_sum, a_machines)`：`machine_allocation.py:266-298`，把数字差异翻译成文字。

**差异行的解读**：
- **正值**：`机台不够 +N` —— M 超过了矩阵 A 的物理上限，意味着算法兜底突破了上限。
- **负值**：`空余 -N` —— 矩阵 A 给的机台数超过算法实际需要的，有富余。
- **0**：刚好顶到上限或刚好匹配。

## 8. 常见调试场景

**Q1：表3 / 表4 没出来？**
检查 `solved_result.get("machine_df")` 是否为 `None` 或空 DataFrame。Excel 导出器在 `excel_exporter.py:145, 162` 处会跳过空数据。

**Q2：某工序机台数突破矩阵 A 上限了？**
看 `compute_actual_machines` 第 4 步兜底逻辑。通常是该工序 Cj 太低 / 矩阵 A 上限给得太小。改 `PROCESS_CAPACITIES` 或 `MATRIX_A` 对应位置。

**Q3：表4 同一日期 M 汇总 ≠ A 理论？**
有两种情况：
- 当天**只有一个订单**：M = A，完美对齐（前提是 Cj 跟矩阵 A 已校准过）。
- 当天**多订单拆分**：每个订单独立查矩阵 A，加起来通常会大于 A(L=总产线数)，因为矩阵 A 是次可加的。这是真实拆单加成，不是 bug。

**Q4：丝网 ≠ 占用产线数？**
现在不应该出现。如果出现，检查 `machine_allocation.py:107` 是不是被改了。
